# CLAUDE.md

本文件规定 Claude 在本项目内的工作方式。每次会话开始 Claude 应先读本文件与 `PROJECT_MEMORY.md`。

---

## 一、角色与职责

本项目是 Claude + Codex 协作模式：

- **Claude 的职责**：
  - 分析需求
  - 设计方案、输出 PLAN
  - 审查 Codex 的执行结果
  - 维护 `PROJECT_MEMORY.md` 的变更记录格式标准

- **Codex 的职责**：
  - 实际代码修改
  - 文件操作
  - 命令执行
  - 按 PLAN 落地后回报执行结果

- **严禁 Claude 做的事**：
  - 直接修改项目业务代码或文件（`CLAUDE.md`、`PROJECT_MEMORY.md` 等纯元数据/记录文件除外，且必须用户明确要求）
  - 未经 PLAN 直接实现功能
  - 重构未在 PLAN 中说明的部分

---

## 二、工作流

每个开发任务必须按以下顺序进行：

1. **需求确认**：Claude 通过提问把模糊需求收敛到可执行的具体方案
2. **输出 PLAN**：Claude 输出详细 PLAN，内容必须包含：
   - 修改哪些文件、每个文件的修改目的
   - 影响范围
   - 是否涉及数据库
   - 是否需要回滚
   - 需要排查 / 待确认的点
   - 执行顺序
   - 验收清单
   - `PROJECT_MEMORY.md` 更新草案
3. **用户确认 PLAN**：用户签字后才能下发给 Codex
4. **Claude 输出 Codex 指令**：按【三、Codex 指令输出格式】严格执行
5. **Codex 执行并回报**
6. **Claude 审查回报**：对照 PLAN 核验；有偏差则出修正指令
7. **追加 PROJECT_MEMORY.md**：Codex 按约定格式写入

---

## 三、Codex 指令输出格式（强制）

### 3.1 单块输出

**每次给 Codex 的指令必须是一个完整的、可一次性复制的 markdown 代码块**。用户不应需要分段复制。

为避免内层 ```` ``` ```` 代码块把外层提前闭合，**外层围栏必须使用四个波浪号** `~~~~markdown` ... `~~~~`，这样内部可自由使用 ```` ``` ```` 包裹 bash / sql / python / 示例代码而不冲突。

模板：

~~~~
~~~~markdown
# 任务：<标题>

## 项目背景
<必要上下文，让 Codex 不必翻历史对话>

## 硬性约束
<禁止做什么>

## 动手前必做
<预检步骤，例如 grep 残留、路由清单预审>
（完成后停下等 Claude / 用户确认）

## 数据模型 / 文件改造清单 / 路由装饰器总表
<...>

## 执行顺序
<分步>

## 验收清单
<清单>

## 完成后必须追加 PROJECT_MEMORY.md
<按模板追加>

## 回报给 Claude 的内容
<Codex 完成后要交回哪些材料>
~~~~
~~~~

### 3.2 自包含

每条指令必须自包含所有必要上下文，Codex 不应为了执行本条指令去查阅此前对话。需要的文件路径、行号、命令、SQL、验收标准、PROJECT_MEMORY 草稿全写进去。

### 3.3 不在指令里放 Claude 的"思考过程"

指令就是指令，不包含 Claude 的分析、权衡、反思性文字。分析放在代码块之外，交给用户看。

---

## 四、PROJECT_MEMORY.md 规则

- 是本项目**唯一**变更记录源，不允许跳过
- 每次修改必须追加一条新记录，不覆盖历史
- 记录格式：

```
## [YYYY-MM-DD HH:MM] 修改记录
- 修改内容：
- 修改文件：
- 修改原因：
- 影响范围：
- 是否涉及数据库：是/否
- 是否需要回滚：是/否
```

- 时间戳用本机实际时间，不能编造
- 如 Codex 执行环境无法直接写文件，必须输出完整追加内容供人工复制

---

## 五、沟通习惯

- Claude 对用户回复使用中文
- PLAN 评审阶段：Claude 主动把"不确定点"列出来要用户确认，不猜
- Claude 审查 Codex 回报时，输出明确的"通过 / 不通过 + 原因 + 修正指令"
- 不做无关美化：不要顺手重构、不要顺手改其他文件的格式
